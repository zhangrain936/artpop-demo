# ArtPop 艺术交易平台 - 技术架构与功能说明

## 1. 技术架构 (Technical Architecture)

本项目目前采用 **前后端全栈单体架构 (Full-Stack Monolithic Architecture)**，基于 Node.js 运行。主要技术栈如下：

### 前端技术栈 (Frontend)
- **框架:** React 18 + Vite (使用 TypeScript 编写)
- **路由:** React Router v6 (`react-router-dom`)，主要采用移动端 H5 的视图和路由设计，包含了统一的底部 Tab 栏。
- **状态管理:** React Context API (`AuthContext` 用于管理用户登录、角色状态、购物车数据等)
- **样式与 UI:** 
  - **Tailwind CSS:** 用于核心响应式样式和原子化控制。
  - **Lucide React:** 全局使用的 SVG 图标库。
- **布局设计:** Mobile-First 设计理念，专门适配移动端分辨率和交互体验。

### 后端技术栈 (Backend)
- **框架:** Express.js (Node.js)，通过 `tsx` 运行。
- **中间件:** 
  - Express JSON Parser (`express.json()`)
  - 开发环境下集成了 Vite Middleware (`createViteServer`) 实现在同一个端口 (3000) 下前后端同构。
- **数据存储 (模拟):**
  - 当前由于并未连接真实的云数据库，使用 **后端内存变量 (`db` 对象)** 作为核心数据源来模拟数据库操作，包含了 `users` (包含买家、学生创作者、管理员)、`products` (艺术作品) 以及购物车缓存信息。

### 数据流向 (Data Flow)
1. **客户端请求:** 从 React 前端发送 RESTful `fetch` 请求到对应的 `/api/*` 后端路由。
2. **鉴权机制:** 前端登录后会获得一个简单的凭证（当前为单纯的 token 字符串标识），在请求时带在 Authorization Header 里。
3. **服务端处理:** Express 后端收到请求后，从模拟内存中进行增删改查。
4. **返回值解析:** 返回 JSON 格式数据供前端渲染。

## 2. 已实现功能清单 (Implemented Features)

### 📌 用户角色与鉴权
- **统一登录页 (`MobileLogin`)**: 支持买家、创作者 (学生卖家)、管理员三种角色的快速模拟登录。
- **身份验证 (`AuthContext`)**: 统一维护全局 User 信息、登录登出操作、购物车计数等。

### 📌 消费者端 (买家功能)
- **首页瀑布流 (`MobileHome`)**: 展示热门、最新上传的美院学生艺术作品，包含顶部推荐 Banner。
- **发现与分类 (`MobileCategory`)**: 多维度筛选与标签点击组，包含“央美毕业季”特色专区，支持多选并实时刷新作品瀑布流。
- **搜索页 (`MobileSearch`)**: 支持按关键词、分类检索艺术商品。
- **商品详情页 (`MobileDetail`)**: 展示单品详细大图、参数（尺寸、年份、包装与发送方式等）、包含动态收藏和加购功能，并呈现运费模式展示（包邮/到付）。
- **艺术家主页 (`MobileAuthor`)**: 展示艺术家个人资料、历史售卖量、个人作品集。
- **购物车与结算的模拟 (`MobileCart`)**: 商品添加到购物车，合计金额，简单结算流程体验。
- **个人中心与订单 (`MobileProfile`&`MobileEditProfile`)**: 买家编辑资料、查看收藏池 (`MobileFavorites`)，功能入口（我的收货地址/客服等展示层）。

### 📌 创作者端 (学生卖家功能)
- **认证与入驻 (`MobileVerify`)**: 买家或新生可提交入驻申请。
- **工作台/上架管理 (`MobileWorkspace`)**: 
  - 学生管理自己的作品，进行上下架编辑与发布。
  - **支持编辑价格:** 可视化抽佣说明（预计平台佣金10%，预计收入90%）。
  - **运费控制**: 编辑作品时支持自主选择“包邮”或“顺丰到付”。
- **IAA 激励广告加速审查功能 (`handleBoost`):** 引入类似前端看激励视频广告为作品赋权的虚拟逻辑。
- **订单管理模块 (`MobileSellerOrders`)**: 创作者可查看买家购买记录，并进行**去发货 (ship)**和**退款 (refund)**等操作的标记。
- **钱包与提现 (`MobileWithdraw`)**: 查看个人收益余额，并提交提现请求到指定的支付宝等账户（前端模拟）。

### 📌 管理员审核后台 (PC端视图为主)
- **商品审核 (`AdminReview`)**: 管理员角色的独立页面，支持查看所有“待审核”商品，并进行“通过”或“拒绝”操作。

## 3. 待实现功能清单 (Pending / To-Do List)

> 本系统目前基于轻量化构建，仍存在大量待完善体验与实际可对接的功能：

### 🛠 后端与存储升级
- **接入真实数据库:** (如 Firebase Firestore / Supabase / MongoDB 等)，替换当前内存模拟的 `db`。
- **接入真实对象存储:** 目前图片的上传为“前端回显”加“Mock地址”模拟，亟需接入真实的云存储（S3/阿里云 OSS/Firebase Storage）实现图片持久化。

### ⚙️ 核心交易流程闭环
- **真实支付接入:** 接入微信支付/支付宝小程序支付或 Stripe 等聚合支付通道，取代目前的虚拟结算按钮。
- **订单状态机补全:** （买家端展示“待支付/待发货/待收货”，卖家端关联物流单号并更新对应状态，买库端能够主动申请逆向退款流程等）。
- **收货地址管理:** 补全买家“管理常用地址列表”页面与下单页面的地址选择功能。

### 💬 社交与IM体系
- **实时消息模块 (`/messages` 页):** 引入站内信或 WebSocket（如 Socket.io），实现买卖双方的在线实时沟通和出价讨论。
- **更强的社区属性:** 评论系统、关注拉新动态 Feed 流功能（用户关注某艺术家后在其时间线上收到上新通知）。

### 🔐 真实鉴权与安全
- **基于 OAuth 或 SMS 的登录注册体系:** 替换目前的直接切号模拟 UI，加入真实验证码通道及 Token (JWT) 加密验证。
- **敏感词过滤系统:** 在上架信息和聊天内过滤不当言论。

### 🎨 前端细节与易用性
- **图片压缩处理:** 移动端在上传证件或画作照片时，前端应加入 canvas 压缩，提升交互体验及减少带宽耗竭。
- **统一异常处理与边界路由:** 例如 404 页面的收敛、网络断开与接口异常降级的骨架屏等。
